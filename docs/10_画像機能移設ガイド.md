# 画像機能移設ガイド

## 1. このドキュメントの目的

このドキュメントは、あるプロジェクトで実装済みの画像アップロード機能を、別プロジェクトへ移設するための設計ガイドです。  
特定の画面名、DB 名、ファイル名には依存せず、「移設先で何を決め、どの責務を分け、どこを検証すべきか」に絞って整理します。

対象は、次のような機能です。

- プロフィール画像
- カバー画像
- 記事や商品に紐づく画像
- ギャラリー画像

固定配置の静的画像は対象外です。

## 2. まず決めるべきこと

移設前に、最低でも次の項目を決めてください。

### 2.1 画像の用途

用途ごとに、別の制約を持たせるべきかを先に決めます。

例:

- `avatar`
- `cover`
- `gallery`
- `og`

用途ごとに分ける理由は次のとおりです。

- 許容サイズが異なる
- アスペクト比が異なる
- 表示先によって必要品質が異なる
- 削除条件が異なる場合がある

### 2.2 出力形式

原則として、保存形式は統一した方が移設しやすくなります。

推奨:

- 保存形式: `webp`
- 入力形式: `jpeg`, `png`, `webp`
- 必要なら `heic` / `heif` を追加

保存形式を統一する利点:

- 表示実装が単純になる
- サムネイル生成や再検証がやりやすい
- content-type の検証がしやすい

### 2.3 サイズ制約

次の 3 つは用途ごとに固定してください。

- 元ファイルの最大サイズ
- 保存後の最大 width / height
- 画質

例:

| 用途    | 元ファイル上限 | 保存後の最大サイズ | 形式 |
| ------- | -------------- | ------------------ | ---- |
| avatar  | 20MB           | 800x800            | webp |
| cover   | 20MB           | 1600x900           | webp |
| gallery | 20MB           | 2000x2000          | webp |

### 2.4 アップロード方式

移設時の設計判断で最も重要なのが、どちらの方式を採るかです。

1. サーバー経由アップロード
2. ストレージ直アップロード

使い分け:

- 小規模、管理画面中心、構成を単純にしたいならサーバー経由
- 本番で外部ストレージを使い、大きいファイルも扱うなら直アップロード

### 2.5 画像参照の持ち方

移設先の DB で、画像をどう参照するかを決めます。

選択肢は主に 2 つです。

1. エンティティが画像 URL を直接持つ
2. `media_assets` のような画像テーブルを作り、ID 参照する

使い分け:

- 1件のエンティティに少数画像なら URL 直持ちで十分
- 画像の再利用、監査、派生管理が必要なら画像テーブル化

## 3. 移設時にも残すべき責務分割

画像機能は、次の 5 層で分けると移設しやすくなります。

| 層                 | 役割                                             |
| ------------------ | ------------------------------------------------ |
| 画像設定           | 用途、許可形式、サイズ、保存パス規約             |
| クライアント前処理 | 入力検証、HEIC 変換、リサイズ、`webp` 化         |
| アップロード層     | サーバー送信またはストレージ直アップロード       |
| サーバー検証層     | content-type、サイズ、保存先、参照整合性の再検証 |
| 永続化層           | DB 反映、旧画像削除、孤児画像の掃除              |

この分割を崩すと、移設時に画面ロジックと画像ロジックが密結合になります。

## 4. 推奨アーキテクチャ

### 4.1 共通モジュール

移設先でも再利用しやすいのは次の単位です。

- `image-config`
  - 用途一覧
  - 許可形式
  - 最大サイズ
  - 保存パス規約
- `image-client`
  - 入力形式判定
  - HEIC 変換
  - リサイズ
  - `webp` 変換
  - 送信前検証
- `image-storage`
  - 保存
  - 削除
  - ストレージ driver 切り替え
  - 管理対象 URL 判定
- `image-upload-api`
  - 直アップロード用トークン発行
  - 完了後の再検証
  - 失敗時 cleanup
- `image-reference-service`
  - 参照中画像の検索
  - 削除可能判定

### 4.2 画面固有モジュール

画面ごとに持つべきなのは次だけです。

- hidden input 名
- フォームの必須項目検証
- どの画像用途を使うか
- DB 更新時にどのカラムへ入れるか

画像の変換ロジックや URL 検証ロジックは、画面に置かない方が移設しやすいです。

## 5. 保存パス規約

保存パスは必ず規約化してください。

推奨形式:

```text
media/<scope>/<entityId>/<timestamp>.webp
```

例:

```text
media/avatar/user-123/1700000000000.webp
media/cover/article-456/1700000000001.webp
```

この規約が必要な理由:

- URL だけで用途判定ができる
- 所有者判定ができる
- 他人の画像 URL を差し込む攻撃を防ぎやすい
- 旧画像と新画像の衝突を避けられる

もし新規作成時点で `entityId` が未確定なら、次のどちらかを採ってください。

1. 先に `draftId` を発行する
2. 一時アップロード用 `uploadSessionId` を使い、保存確定時に正規パスへ移す

## 6. アップロードフロー

### 6.1 サーバー経由アップロード

フロー:

1. クライアントで入力形式とサイズを検証する
2. 必要なら HEIC を変換する
3. クライアントでリサイズし `webp` 化する
4. 変換済みファイルをフォームまたは API でサーバーへ送る
5. サーバーで再度検証する
6. ストレージへ保存する
7. DB に保存先参照を反映する

向いているケース:

- 管理画面のみ
- トラフィックが小さい
- 構成を単純に保ちたい

### 6.2 ストレージ直アップロード

フロー:

1. クライアントで入力形式とサイズを検証する
2. 必要なら HEIC を変換する
3. クライアントでリサイズし `webp` 化する
4. サーバーからアップロード用トークンを取得する
5. クライアントがストレージへ直接アップロードする
6. 完了コールバックでサーバーが再検証する
7. 検証済み URL または asset ID を本保存リクエストへ渡す
8. サーバーで再度参照整合性を確認して DB 反映する

向いているケース:

- 本番で外部ストレージを使う
- 画像サイズが大きい
- アプリサーバーを経由させたくない

### 6.3 直アップロード時に必要な考え方

直アップロードでは、「アップロード済み URL をそのまま信用しない」が前提です。

最低限、次を検証してください。

- URL が自分の管理下ストレージか
- pathname が規約に一致するか
- `scope` と `entityId` が一致するか
- content-type が期待どおりか
- ファイルサイズが上限以下か
- 実際に `webp` として読めるか
- width / height が用途ごとの上限以内か

## 7. クライアント前処理で必要なこと

クライアント側は、次の責務だけを持たせるのが安全です。

- 入力形式の判定
- 元ファイルサイズの判定
- HEIC / HEIF の変換
- リサイズ
- `webp` 変換
- 送信中ステータス管理
- 失敗時の cleanup 呼び出し

クライアント側に置かない方がよいもの:

- 最終的な権限判定
- 保存先の正当性判定
- DB 参照整合性判定

それらは必ずサーバーで再検証してください。

## 8. サーバー側で必ず検証すること

移設先でも、次の検証は省かない方がよいです。

### 8.1 入力ファイル検証

- 許可拡張子
- MIME type
- 元ファイルサイズ

### 8.2 最適化後ファイル検証

- format が `webp`
- width / height が上限以内
- 画像として読める

### 8.3 保存先検証

- pathname が規約どおり
- `scope` が一致
- `entityId` が一致
- 期待しない host を拒否

### 8.4 参照検証

本保存時には、アップロード済み参照が本当にそのエンティティのものかを確認してください。

これは、他ユーザーが別エンティティ用 URL を差し込むのを防ぐためです。

## 9. 削除と cleanup の設計

画像機能の移設で壊れやすいのは、アップロードより削除です。

ルールは次の 3 つに分けてください。

### 9.1 保存失敗時

今回アップロードした画像だけを削除する

### 9.2 更新成功時

旧画像と新画像が異なる場合だけ、旧画像を削除する

### 9.3 cleanup API

フォーム送信失敗やキャンセル時は、一時アップロード済み画像を掃除する  
ただし削除前に、まだ DB 参照が残っていないか確認する

重要なのは順序です。

- 旧画像削除は DB 更新成功後
- 新画像 cleanup は DB 更新失敗時のみ

この順序を崩すと、更新失敗時に画像だけ失う事故が起きます。

## 10. データモデルの選び方

### 10.1 URL 直持ち

向いているケース:

- 1エンティティに 1〜数枚
- 再利用しない
- 管理画面中心

利点:

- 実装が単純
- 移設しやすい

欠点:

- 画像監査が弱い
- 派生画像管理がしづらい

### 10.2 画像テーブル化

向いているケース:

- 画像の再利用がある
- 作成者や更新履歴を持ちたい
- サムネイルや複数 variant を管理したい
- 画像単位で公開状態や審査状態を持ちたい

保持するとよい項目:

- `id`
- `scope`
- `ownerType`
- `ownerId`
- `storageKey`
- `publicUrl`
- `format`
- `width`
- `height`
- `fileSize`
- `status`
- `createdBy`
- `createdAt`

## 11. 依存ライブラリの考え方

移設時に必要な依存は、役割で選んでください。

### 11.1 必須に近いもの

- 画像最適化ライブラリ
  - 例: `sharp`
- 外部ストレージ SDK
  - 利用するストレージに応じて選択

### 11.2 条件付き

- HEIC 変換ライブラリ
  - iPhone 画像を受けるなら必要
- `.env` 読み込みライブラリ
  - 実行環境による

重要なのはライブラリ名ではなく、次の責務が満たせることです。

- 回転補正
- リサイズ
- `webp` 出力
- metadata 読み取り
- ストレージ保存
- ストレージ削除

## 12. 移設チェックリスト

### 12.1 設計

- 画像用途を列挙した
- 用途ごとの最大サイズを決めた
- 保存形式を決めた
- アップロード方式を決めた
- 保存パス規約を決めた
- DB の参照方法を決めた

### 12.2 実装

- クライアント前処理を共通化した
- サーバー再検証を実装した
- cleanup API を用意した
- 旧画像削除の順序を定義した
- 参照中画像を削除しない仕組みを入れた

### 12.3 テスト

- 許可形式テストがある
- サイズ上限テストがある
- 保存パス規約テストがある
- `webp` 最適化テストがある
- 不正 URL 拒否テストがある
- 更新時の旧画像削除条件テストがある

## 13. 実装順序

移設は次の順で進めると安定します。

1. 用途、サイズ、保存形式を決める
2. 保存パス規約を決める
3. サーバー側の保存・削除 API を作る
4. サーバー側の検証を先に実装する
5. クライアント前処理を載せる
6. 画面へ組み込む
7. cleanup を入れる
8. 更新時の旧画像削除を入れる
9. テストを書く

先に画面から作ると、後で保存規約や cleanup 条件の変更が大きくなりやすいです。

## 14. 移設時によくある落とし穴

- 入力形式だけ見て MIME type を見ない
- クライアントで変換した結果をサーバーで再検証しない
- 直アップロード URL をそのまま信用する
- 旧画像を DB 更新前に消す
- cleanup で参照中画像まで消す
- `entityId` 未確定のまま保存パス規約を決めていない
- 用途ごとの上限サイズを持たず、全画像を同じ制約で扱う

## 15. 最小構成の結論

別プロジェクトへ持っていくなら、最低限必要なのは次の 6 点です。

1. 用途ごとの画像設定
2. 保存パス規約
3. クライアント前処理
4. サーバー側の再検証
5. DB 反映後の旧画像削除ルール
6. 参照中画像を守る cleanup

この 6 点が揃っていれば、画面やフレームワークが変わっても画像機能は移設しやすくなります。
