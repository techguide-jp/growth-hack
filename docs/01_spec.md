# 仕様書ドラフト

## 1. 概要

### 1.1 サービス目的
ハッカソンで生まれた作品（アプリ/ツール）の情報を集約し、コメント・レビュー・応援・仲間作り（DM/プロジェクト会話）と、**進捗/相談が流れるタイムライン**により、個人・小規模チームの**継続開発を後押し**する。

### 1.2 コア価値（体験の柱）
- **作品の発見**: イベント/タグ/検索/ソート
- **継続の可視化**: 更新投稿（リリースノート）・進捗投稿
- **助け合い**: 相談投稿（固定テンプレ）・ベスト回答・DM/プロジェクト会話
- **称賛/応援**: リアクション（5種）・レビュー・支援記録（外部投げ銭の記録）
- **運営の周知**: イベント作成/告知配信（管理者）

### 1.3 スコープ（MVP）
- フレームワーク: **SvelteKit**
- DB: PostgreSQL想定
- 画像: スクリーンショット（オブジェクトストレージ）
- 通知: アプリ内通知 + **即時メール**
- 投げ銭: 外部リンク（決済は行わない） + **支援記録（受領側確定）**
- コミュニティ: タイムライン（進捗/相談/見せびらかし）・コメント・リアクション（5種）
- メッセージ: 個人DM（1対1）+ project会話（グループ）

### 1.4 KPI（仮）
- 投稿プロジェクト数 / 公開プロジェクト数
- 更新投稿（リリースノート）数、進捗投稿数
- 相談投稿の解決率（solved率）、ベスト回答数
- DM開始数 / project会話アクティブ数
- 支援確定件数 / 確定支援額（JPY）

---

## 2. ユーザと権限

### 2.1 ロール
- `user`: 全ユーザ共通（投稿も応援もできる）
- `admin`: 運営のみ（イベント/告知 管理）

### 2.2 権限の原則（所有者ベース）
- プロジェクトの編集/公開: **オーナーのみ**（またはメンバーに拡張可能）
- 支援記録の確定（受領処理）: **プロジェクトオーナーのみ**
- 会話の閲覧: **conversation_members のみ**

## 3. 主要機能要件

### 3.1 オンボーディング（フォーカスUI）
- 初回ログイン時に「何をしたいか（複数選択）」
  - 作品を投稿したい / 作品を探して応援したい / 仲間を見つけたい / イベント情報を追いたい
- 選択結果は `user_preferences.focus_modes` に保存
- ホーム/ダッシュボードのセクション順、強調カード、CTA（行動ボタン）を出し分け
- 設定画面からいつでも変更可能

### 3.2 プロジェクト（作品）
- ユーザはプロジェクトを複数作成できる
- プロジェクトの基本情報
  - タイトル / サマリ / 詳細説明
  - 公開URL / リポジトリURL / デモURL
  - タグ
  - スクリーンショット（複数）
- 状態
  - 下書き（draft）/ 公開（published）/ アーカイブ（archived）
- イベントとの紐づけ
  - `project_events` により多対多

### 3.3 更新投稿（リリースノート/お知らせ）
- プロジェクト単位で更新投稿を作成
- 更新投稿はプロジェクト詳細で時系列表示
- **更新投稿はタイムライン/アクティビティに自動的に流す**（ひと気を作る）

### 3.4 コミュニティタイムライン（ワイワイ感の核）
- 投稿タイプ（MVP固定）
  1) **進捗**: 今日やったこと/次やること
  2) **相談**: 困りごと（固定テンプレあり）
  3) **見せびらかし**: できた！の共有（URL/スクショ誘導）
- 公開範囲: MVPは **全公開**
- フィルタ（タブ）
  - 全体 / フォロー中 / イベント別 / プロジェクト別
- 相談投稿のテンプレ（固定）
  - 状況 / 困りごと / 試したこと / 環境
- 相談は解決済みにできる
  - コメントから「ベスト回答」を選択（投稿者のみ）
  - `status=open/solved` + `accepted_comment_id` を保持

### 3.5 コメント（汎用）
- コメント対象
  - プロジェクト / 更新投稿 / タイムライン投稿
- 返信（スレッド）: MVPでは **フラット**でも可（将来スレッド化余地）

### 3.6 リアクション（5種類）
- 対象
  - プロジェクト / 更新投稿 / タイムライン投稿 / コメント（MVPは投稿のみでも可）
- 5種類（案：後でラベル変更可）
  1. 👏 拍手
  2. 👍 いいね
  3. 💡 なるほど
  4. 🔥 アツい
  5. 🛟 助ける
- 1ユーザにつき、対象×種類ごとに1回（ユニーク制約）

### 3.7 フォロー
- フォロー対象: ユーザ / プロジェクト
- フィード（フォロー中タブ）や通知に利用

### 3.8 支援（外部投げ銭 + 支援記録）
- 決済は外部
  - プロジェクトに外部支援リンクを登録（例: GitHub Sponsors, BuyMeACoffee など）
- 支援記録（JPY固定/期限なし）
  - 支援者が外部で支援した後に、アプリで「支援記録」を作成
  - 支援記録作成直後: `status=awaiting_owner`
  - 受領側（プロジェクトオーナー）が「受け取りました」で `status=confirmed`
  - 表示・集計は **confirmed のみ**
  - 確定前のみ
    - 支援者キャンセル: `cancelled`
    - 受領側却下: `rejected`
  - 却下理由入力: MVP不要
- 重要操作は `audit_logs` へ記録

### 3.9 仲間募集・参加申請
- プロジェクトに「仲間募集（help_wanted）」を作成
  - 欲しい役割/スキル/内容、ステータス（open/closed）
- 参加申請（join_requests）
  - 申請→オーナーが承認/却下

### 3.10 メッセージ（サービス内）
- 個人DM（1対1）
  - 既存の相手とのDMスレッドは再利用（重複作成しない）
- project会話（グループ）
  - プロジェクトメンバーが参加
  - 参加者制御: `conversation_members`
- 添付: MVPではなし（テキストのみ）
- 未読管理: `message_reads.last_read_at`（簡易でOK）

### 3.11 通知（アプリ内 + メール即時）
- 通知チャネル
  - アプリ内: `notifications`
  - メール: `email_outbox` に積み、ワーカーで送信（即時）
- 通知設定
  - 種類×チャネル（アプリ内/メール）をユーザがON/OFF
  - 将来のダイジェスト設定（daily/weekly）は設定項目だけ先に用意（MVPは未実装でもOK）

## 4. 画面要件（概要）
- `/onboarding` フォーカス選択
- `/` ホーム（タイムライン＋最近の動き＋最近更新作品 等。フォーカスで並び替え）
- `/timeline` タイムライン（全体/フォロー/イベント/プロジェクト）
- `/events` `/events/[id]`（イベント詳細に「ルーム（イベントタイムライン）」）
- `/projects`（検索/タグ/ソート切替、デフォルト更新順）
- `/projects/[id]`（概要/更新/レビュー/コメント/仲間募集/応援/会話）
- `/dashboard`（自分の作品、承認待ち支援、未読DM、自分の未解決相談）
- `/messages` `/messages/[id]`
- `/settings`（フォーカス/通知設定）
- `/admin`（イベント・告知）

---

## 5. 非機能要件

### 5.1 パフォーマンス
- 一覧はページネーション（20件程度）
- 検索/ソートはインデックス設計（更新順がデフォルト）

### 5.2 セキュリティ
- 認証: OAuth/メール（SSO）
- 権限: 所有者・参加者のサーバ側チェック
- DM/タイムライン/コメント: レート制限

### 5.3 運用
- ログ/監視/エラートラッキング
- メール送信失敗のリトライ（outbox再送）

### 5.4 法務
- 利用規約/プライバシーポリシー
- 支援記録は「決済機能ではなく記録である」旨の明記

## 6. リスク & バッファ
| リスク | 確率 | 影響度 | 緩和策 |
|---|---:|---:|---|
| タイムライン/DMでスパム・不快行為が発生 | 中 | 高 | レート制限、`is_hidden` など将来の非表示フラグ、運用ルール整備（後でブロック追加） |
| タイムラインが過疎ると逆に寂しい | 中 | 高 | 更新投稿/アクティビティを混ぜる、テンプレで投稿しやすく、イベントルームで密度を上げる |
| 支援記録が自己申告で信頼性が揺らぐ | 中 | 中 | 確定は受領側のみ、確定のみ集計、監査ログ |
| メール通知が多すぎる | 中 | 中 | 種類×チャネル設定、初期は重要通知だけメールON |

- 見積りバッファ推奨: **+20%**（機能横断が多いため）
